# Blueprint 到 Godot Demo JSON 生成器可行性与验证报告

时间：2026-06-19 14:50:47

## 结论

自研 `blueprint -> Godot demo JSON` 生成器可行，当前项目已经达到“六章类银河城灰盒地图自动生成 + Godot 可加载验证”的阶段。

难度不在 JSON 转换本身，而在三层质量控制：

1. 地图拓扑：主线、支线、能力门、捷径、回环、保存点、Boss runback 的结构合理性。
2. 物理可玩：平台间距、跳跃可达、冲刺可达、危险区和敌人布置不造成软锁。
3. 关卡手感：房间节奏、视野引导、主题辨识度、战斗压力曲线、可读性和探索奖励。

因此如果目标是“丝之歌类型”的地图，不建议复刻具体商业地图；应抽象其通用设计原则：多层垂直空间、区域主题强区分、能力回流、捷径打开后的路径压缩、短而紧的 Boss 回程、主线和支线交织。

## 难度评估

| 阶段 | 难度 | 是否能由我自动完成 | 说明 |
| --- | --- | --- | --- |
| Blueprint schema | 中 | 可以 | 定义区域、房间、门、能力、捷径、奖励、Boss 节点 |
| JSON 转换器 | 低到中 | 可以 | 当前已有基础实现，继续扩字段即可 |
| 灰盒可玩地图 | 中 | 可以 | 需要持续验证跳跃距离、门锁顺序、回路和软锁 |
| 丝之歌式结构感 | 中高 | 可以做到原型级 | 需要规则库表达垂直空间、回环密度、主支线节奏 |
| 商业级关卡手感 | 高 | 不能完全 0 人工 | 需要人审美术节奏、战斗压力、空间记忆点 |
| 正式 TileMap 美术落地 | 高 | 可半自动 | 可生成初版，但最终打磨仍建议人工审阅 |

## 下一步最短交付路径

1. 把当前 runtime 的单个 `shortcut_open` 升级为 `opened_shortcuts` 字典，让每个捷径能独立打开和保存。
2. 做 Godot 游戏内 debug overlay，直接显示当前房间、能力门、捷径状态、可达性和章节 ID。
3. 增加 room archetype 规则库：竖井、竞技场、补给间、Boss 前厅、支线奖励间、能力教学间、捷径门厅。
4. 再做 TileMap/主题化模板，把灰盒平台替换成可视化场景块，而不是直接追求一次生成最终美术。

## 风险

- 当前是灰盒 JSON，不是正式 TileMap 生产线。
- 地图结构可验证，但“好不好玩”还需要玩家路径模拟、实机跑图和人工抽查。
- 不能直接复制《丝之歌》的地图布局，只能生成同类型设计语言的原创地图。
